# 견적 → 프로젝트 문서화 표준 SOP (v1.0)

quote-maker에서 견적 검토 후 노션에 프로젝트 페이지를 만들고 개발/배포까지 가는 **공통 워크플로우 표준**.
거의 모든 프로젝트(렌탈/행사/판매)에 일관 적용한다.

> 이 문서는 작업 환경(로컬, CI, 다른 PC, 다른 팀원)에 관계없이 동일한 절차로 진행되도록 GitHub에 보관한다.

---

## 1. 단계 책임 매트릭스

| # | 단계 | 트리거 | 담당 | 산출물 | SLA |
|---|------|--------|------|--------|-----|
| 1 | 견적 폼 입력 | 사업팀 요청 | 사업팀 | Quote (status: pending) | — |
| 2 | 견적 자동 산출 | 폼 제출 | 시스템 | 견적 초안 (pending → reviewing) | 즉시 |
| 3 | **개발팀 검토** | 검토 시작 | 개발팀 | 댓글 회신 (옵션 제시) | **영업일 1일** |
| 4 | 사업팀↔고객 합의 | 댓글 응답 | 사업팀 | A/B 옵션 선택 또는 수정 요청 | 영업일 2일 |
| 5 | **승인 게이트** | 승인 버튼 | 개발팀 | 자동 검증 통과 | 즉시 |
| 6 | **노션 페이지 자동 생성** | 승인 이벤트 | 시스템 | 메인 + 하위 5종 스켈레톤 | 즉시 |
| 7 | 설계 문서 작성 | 페이지 생성 후 | 개발 담당 | 하위 5종 본문 | D-7 까지 |
| 8 | 개발 착수 | 설계 완료 | 개발 담당 | status → 개발 진행 | — |
| 9 | 배포/현장 | 납기일 | 개발+사업 | status → 행사 진행 중 | 납기일 |
| 10 | 회수/정산 | 행사 종료 | 사업팀 | status → 완료 | 행사 +7일 |

## 2. 노션 페이지 표준 구조 (v1.1 통일 양식)

**위치**: 개발팀 워크 스페이스 → **진행 중 프로젝트** DB
DB ID: `199900bd-0a45-806f-85bd-000b55a42e5b`
**기본 템플릿**: [프로젝트 템플릿](https://app.notion.com/p/1a7900bd0a458076ad94e0dd81ca127b) — 하위 스켈레톤 0~6 포함, duplicate해서 시작

### 2.0 보안 룰 (필수)

- **견적 금액·항목 단가·계약금 환산액은 노션에 쓰지 않는다.** 다른 팀원도 열람 가능한 공간이므로 민감 정보 금지.
  - 기재 허용: 견적번호(QT-…), 유형, 상태, 항목 "이름", quote-maker 링크
    (견적 URL은 비로그인 접근 차단 확인됨 — 링크 자체는 안전)
  - 금액 자리는 `quote-maker 참조 (로그인 필요)`로 대체
  - 계약 조건은 40%/60% **비율까지만** — 환산 금액 기재 금지
  - DB 속성(비고)에도 금액 금지 — `견적 QT-…` 형식만
- **할일 체크리스트는 노션 프로젝트 페이지에 두지 않는다** (주간목록으로 관리)

### 2.1 메인 페이지 — 한 줄 요약 + 하위 링크만

메인에 본문 섹션을 두지 않는다. 기본정보/견적/특이사항 포함 모든 내용은 하위 페이지에 작성한다.

```
《한 줄 요약 — 행사명(장소) / 장비 / 핵심 기능 / 특이사항》
---
<page>0. 기본 정보 및 견적</page>
<page>1. 기능 명세서 (Feature Spec)</page>
<page>2. 아키텍처 및 기술 스택</page>
<page>3. [도메인 문서]</page>
<page>4. 개발 일정</page>
<page>5. 사전 확인 및 협의 사항</page>
<page>6. 특이사항 및 리스크</page>
<page>.env</page>   ← 필요 시
```

### 2.2 DB 속성 자동 설정
| 속성 | 초기값 |
|------|--------|
| Status | `문서 작업` |
| 행사 날짜 | 견적의 행사일(시작~종료) |
| 개발 종료 및 배포 | 견적의 납기일 |
| 담당자 | 견적 요청자 |
| 개발 담당 | 개발 담당자 |
| 비고 | `견적 {QT-번호}` (금액 금지) |

### 2.3 하위 페이지 구성 — 0~6 고정, 3번만 도메인별 제목 변경

| # | 문서 | 핵심 내용 |
|---|------|-----------|
| 0 | 기본 정보 및 견적 | 행사/장비/납기 + 견적번호·상태·항목 이름·quote-maker 링크 (금액 X) |
| 1 | 기능 명세서 | 사용자 플로우, 화면별 기능, 예외 처리 표, 비기능 요건 |
| 2 | 아키텍처 및 기술 스택 | 재사용 방침·변경점 표, 시스템 구조도, 기술 스택, 배포 |
| 3 | [도메인 문서] | 카드프린터→**카드 디자인 및 인쇄 사양** / 서버 연동→**API 설계** / 특수 장비→**하드웨어 사양** (1~2종) |
| 4 | 개발 일정 | 마일스톤 표, 일정 평가(영업일≥5 게이트 충족 여부), 리스크 표 |
| 5 | 사전 확인 및 협의 사항 | 디자인 소스 / 운영 프로토콜 / 비용 구분 / 인프라 / 계약·정산(비율만) 체크리스트 |
| 6 | 특이사항 및 리스크 | 검토 중 발견한 scope 특이점·타 프로젝트와의 차이 bullet |

세부 골격은 노션 기본 템플릿의 각 하위 페이지 스켈레톤을 따른다.

## 3. 사전 확인 사항 표준 카테고리

### 공통 필수 (3종)
1. **디자인/콘텐츠 소스** — PSD/AI 원본 제공 주체, 폰트/로고 권한
2. **운영 프로토콜** — 장소별 인원/장비 분배, 운영 시간, 처리량
3. **추가 비용 항목** — 장비 렌탈비, 출장비, 현장 지원비, 소모품

### 도메인별 추가
- **카드프린터**: 카드 양식 / 소모품 공급 / 인프라(전원·네트워크)
- **웹/서버**: API 키 / 도메인·SSL / 인증(SSO/OIDC)
- **AI 기능**: API 사용료 / 데이터 보존 정책 / 폴백 모델

### 마지막 공통
- **계약/정산** — 계약서 시점, 계약금 40%/잔금 60% 적용 여부, 세금계산서 조건 (환산 금액은 노션 기재 금지 — 2.0 보안 룰)

## 4. 승인 게이트 — 자동 검증 4종 (구현 예정)

승인 버튼 누를 때 시스템이 검증, 위반 시 **모달 경고 + 무시 사유 입력 강제**.

| # | 검증 | 위반 사례 |
|---|------|-----------|
| 1 | 모든 항목 단가가 마스터 가격표 ±20% 이내 | QR 업로드 100k → 견적 150k (50% 초과) |
| 2 | 렌탈 유형이면 렌탈비 라인 존재 | 렌탈인데 개발비만 잡힘 |
| 3 | 견적 합계 = 댓글에 명시된 금액 | 댓글 90만 / 시스템 93.5만 |
| 4 | 행사일/납기일 사이 영업일 ≥ 5 | 납기 D-2 |

위 4종 모두 [QT-20260420-001 스포티파이 학생증 건](https://quote-maker-alpha.vercel.app/dashboard/quotes)에서 실제 발생한 누락/불일치 사례 기반.
